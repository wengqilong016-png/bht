你现在负责对 B-ht 项目做一次“上线前缺陷审查计划式执行”，目标不是泛泛找问题，而是系统性识别会影响真实用户使用、数据正确性、业务流程、移动端可用性和线上稳定性的缺陷。

# 一、总目标

在不做无关重构的前提下，识别并排序 B-ht 项目中最可能导致以下后果的真实问题：

- 核心流程无法完成
- 数据写错、漏写、重复写
- UI 状态与真实业务状态不一致
- 认证、权限、用户隔离失效
- 离线、弱网、刷新、重进后状态异常
- Web 与移动端行为不一致
- 地图、定位、拍照、上传、同步链路失效
- 测试未覆盖真实高风险路径，导致“看起来没问题，实际上会翻车”

最终目标不是“多提建议”，而是输出一份可执行的问题清单、风险排序和验证方案。

# 二、成功标准

本次审查完成后，必须达到以下结果：

1. 明确列出高风险模块与核心业务链路
2. 找出真正会影响使用的缺陷，而不是风格问题
3. 每个问题都能说明：
   - 发生位置
   - 触发条件
   - 为什么是问题
   - 对用户/业务的影响
   - 修复方向
4. 对高风险但证据不足的问题，明确标注“需进一步验证”
5. 给出修复后的验证方案，而不是只说“建议测试一下”

# 三、非目标

以下内容默认不作为本次重点输出：

- 命名、格式、注释、代码风格
- 与真实故障无关的性能优化建议
- 纯理论上的“也许可以重构”
- 没有证据支撑的猜测性问题

只有当这些内容已经直接导致真实 bug 时，才允许输出。

# 六、执行阶段

## 阶段 1：业务与风险建模

目标：
- 先识别项目的核心业务流程
- 判断哪些流程一旦出错就直接影响上线

必须输出：
- 核心业务流列表
- 每条业务流的关键节点
- 高风险模块清单
- 优先审查顺序

重点回答：
- 用户最核心的操作是什么？
- 哪些流程涉及写库、上传、定位、鉴权、同步？
- 哪些地方一旦出错最难被普通测试发现？

## 阶段 2：入口与状态流梳理

目标：
- 理清从页面入口到数据提交/同步完成的真实调用链
- 找到状态流和数据流断点

必须关注：
- 页面入口
- hooks / context 状态来源
- api / supabase 请求链
- 成功、失败、重试、取消分支
- 本地状态与远端状态的关系

必须输出：
- 核心调用链
- 状态流说明
- 可能的断点或竞态点

## 阶段 3：高风险缺陷识别

目标：
- 只找会影响真实使用的问题
- 不找样式、命名、代码洁癖类问题

重点排查：

1. 认证与权限
   - 登录态恢复
   - token/session 失效后的行为
   - 用户切换后缓存污染
   - 查询是否带正确用户范围

2. 提交流程
   - 重复提交
   - loading/disabled 失效
   - 成功提示早于真实成功
   - 失败后脏状态残留

3. 离线与同步
   - 离线时是否真正入队
   - 联网恢复后是否乱序/重复/漏同步
   - 是否把“待同步”误展示成“已成功”

4. React 状态与异步
   - effect 依赖错误
   - 闭包旧值
   - 请求竞态导致 UI 倒退
   - optimistic update 缺少回滚

5. 原生能力与平台差异
   - 定位权限拒绝
   - 相机/上传在 Capacitor 与 Web 的差异
   - 路径、blob、文件元数据处理不一致

6. 页面恢复能力
   - 刷新后状态丢失
   - 返回后脏数据
   - 深链接/直接进入异常
   - 多步骤流程中断后恢复不正确

## 阶段 4：问题定级与收敛

目标：
- 不堆积问题，要按上线风险排序

定级标准：
- Critical：核心流程失败、数据错误、权限问题、线上事故级
- High：高概率影响真实用户使用
- Medium：边界条件或特定场景明显异常
- Low：真实问题但影响面较小

必须输出：
- 问题优先级排序
- 每个问题的触发条件
- 用户影响范围
- 修复优先级建议

## 阶段 5：修复建议

目标：
- 提供最小、可落地的修复方向
- 不做无关重构建议

每个问题必须给出：
- 最小修复方向
- 涉及的模块
- 可能影响的上下游
- 修复后必须验证的点

## 阶段 6：修复后验证计划

目标：
- 确保修复不是“表面不报错”，而是真恢复业务能力

验证分层如下：

### A. 静态验证
适用：类型、接口、依赖关系、明显语法和导入问题
建议命令：
- npm run typecheck
- npm run lint

### B. 定向单测 / 集成验证
适用：纯逻辑、状态流、数据转换、边界判断
建议：
- 优先跑对应测试文件
- 没有现成测试时，至少说明应补的关键断言

### C. 业务流验证
适用：提交、同步、登录、上传、定位、页面跳转
必须验证：
- 正常路径
- 失败路径
- 重试路径
- 重复点击
- 刷新恢复
- 弱网/断网恢复（如果涉及离线逻辑）

### D. E2E / 真实交互验证
适用：跨页面、多步骤、移动端/浏览器行为差异
建议命令：
- npm run test:e2e
- 或指定最相关的 e2e 用例

### E. 人工场景回归
对以下改动必须做人工确认：
- 认证 / 权限
- 离线同步
- 定位 / 相机 / 上传
- Capacitor 平台行为
- 任何“表面成功、实际失败”风险点

# 七、验证通过标准

一个问题只有在以下条件满足后，才算真正关闭：

1. 代码层面的根因已被处理
2. 相关路径的成功/失败分支都验证过
3. 不会引入重复提交、脏状态、缓存污染或用户隔离问题
4. UI 提示与真实状态一致
5. 涉及离线/同步/原生能力的场景已验证恢复行为
6. 至少有一种可复现方式证明“修复前有问题，修复后无问题”

# 八、失败后的处理规则

如果某个问题修复尝试失败，必须：
1. 回到根因重新分析
2. 说明上一次方案为什么失败
3. 明确这一次方案与上一次的不同点
4. 不允许对同一代码做无意义微调式反复尝试

如果连续 2 次失败，必须暂停修改，重新整理调用链与状态流。

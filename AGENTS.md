# AGENTS.md

## 项目目标
这是一个需要真实可用、可上线运行的项目。
修复问题时不能只“消报错”，目标是恢复真实功能，并通过可复现验证证明修复有效。

---

## 核心原则
- 先理解再修改
- 先根因后修复
- 最小必要改动
- 不重复失败路径
- 不破坏业务换测试通过
- 保持接口与行为一致

---

## 额度与智力平衡规则（核心）
- 限制读取范围，不限制推理深度
- 默认只查看 1~3 个相关文件
- 禁止扫描整个仓库（除非必要且说明原因）
- 每次只解决一个问题
- 禁止“顺便”修改无关内容
- 输出精简，但推理必须完整
- 上下文过长时必须先总结再继续

---

## Agent 工具运行规则
- 搜索文本/调用链时优先使用 `rg`，因为它比 `grep` 更快、更省上下文。
- `rg` 不是每次都需要下载：如果系统或项目环境里已经有 `rg`，直接使用。
- 使用 `rg` 前可先执行 `command -v rg` 判断是否存在。
- 如果 `rg` 不存在，再加载 x-cmd：
  - `. ~/.x-cmd.root/X`
  - `x env use ripgrep`
- `x env use ripgrep` 首次可能需要联网下载；只有在本地缺少 `rg` 且确实需要搜索时才执行。
- 如果网络、权限或沙箱导致安装失败，不得反复重试；改用已有工具，例如 `grep`、`find`、`sed`，并说明降级原因。
- 不得为了方便而安装无关工具、升级依赖或修改项目配置。

## 工作流程（强制执行）

### 阶段 1：问题定义
必须明确：
- 复现步骤
- 预期行为
- 实际行为
- 报错信息

---

### 阶段 2：范围锁定（关键省额度）
- 仅查看：
  - 报错文件
  - 直接调用方
  - 入口文件
- 初始范围 ≤ 3 个文件
- 扩展范围必须说明原因

---

### 阶段 3：根因分析（禁止修改代码）
必须输出：
- 调用链分析
- 1~3 个根因（按概率排序）
- 为什么是这些原因
- 最小修复方案

---

### 阶段 4：最小修复
- 只修改直接相关代码
- 禁止顺手重构
- 禁止扩大范围
- 保持现有逻辑一致

---

### 阶段 5：验证（分级）

#### 默认：最小验证
- 只验证相关功能
- 不跑全量命令

#### 完整验证（仅必要时）
- npm run lint
- npm run test:ci
- npm run build

---

### 阶段 6：失败处理（关键收敛机制）
- 必须分析失败原因
- 不得重复相同方案
- 明确本次方案与上次不同点

---

### 阶段 7：输出（必须包含）
- 根因
- 修改文件
- 修改目的
- 验证命令
- 验证结果
- 未通过原因
- 剩余风险

---

## 验证执行规则（防烧额度）
- 禁止重复运行同一命令
- 若失败且无新改动，不得重跑
- 优先局部验证
- 全量验证必须有理由
- 文档、注释、规则文件改动：默认只做读回验证，不跑 lint/test/build。
- 单文件纯逻辑改动：优先运行对应单测或 `npm run typecheck`，不直接跑全量。
- UI 局部改动：优先启动页面或相关局部测试，只有影响共享组件/路由时才扩大验证。
- 数据库、认证、离线同步、支付/结账、权限相关改动：必须说明风险，并选择针对性测试；必要时再跑完整验证。
- 只有以下情况才考虑完整验证：
  - 修改了共享基础设施、构建配置、依赖、类型定义或全局上下文
  - 局部验证无法覆盖真实风险
  - 准备上线、合并或发布
- 若未跑完整验证，必须明确说明原因，而不是把它写成未解决风险。

---

## 明确禁止
- 跳过验证
- 注释代码假修复
- 删除报错代码
- 无依据改依赖
- 修改无关文件
- 编造结果
- 扫描整个仓库

---

## 收敛规则（极关键）
如果连续 2 次修复失败：
- 必须停止当前思路
- 重新做根因分析
- 说明之前为什么错
- 提出新方案

禁止：
- 微调同一代码反复尝试
- 重复类似修改
